## ADDED Requirements

### Requirement: 系统必须为合格的大文件发送建立极速传输会话

系统 MUST 仅在发送端与接收端已配对、文件大小达到极速阈值且双方能力协商成功时建立极速传输会话。系统 MUST 为该会话分配唯一 `transferId`、`messageId` 和 `sessionId`，并将其纳入统一传输状态机。

#### Scenario: 合格文件进入极速会话
- **WHEN** 用户选择的本地文件达到极速阈值，发送目标为已配对设备，且接收端声明支持极速传输
- **THEN** 系统创建极速传输会话，分配统一的 `transferId`、`messageId` 和 `sessionId`，并通过统一事件流进入 preparing / sending 状态

#### Scenario: 不满足条件时不建立极速会话
- **WHEN** 文件未达到极速阈值，或对端未配对，或任一方不支持极速能力
- **THEN** 系统不得建立极速会话，并必须改走现有普通文件传输路径

### Requirement: 系统必须使用独立高速数据面传输文件字节

系统 MUST 在控制面协商成功后使用独立数据端口承载文件字节流。数据面 MUST 使用会话级鉴权信息，仅允许持有有效 `transferToken` 的连接加入会话。数据面 MUST 支持从单连接启动，并在同一会话中扩展为多连接 striping。

#### Scenario: 合法数据连接加入会话
- **WHEN** 发送端使用有效的 `sessionId`、`transferToken` 和流索引建立数据连接
- **THEN** 接收端必须接受该连接并允许其进入数据帧传输阶段

#### Scenario: 非法数据连接被拒绝
- **WHEN** 发送端使用过期、无效或与会话不匹配的 `transferToken` 建立数据连接
- **THEN** 接收端必须拒绝该连接并不得写入任何文件内容

### Requirement: 系统必须支持自适应 striping 以提升局域网大文件吞吐

系统 MUST 从单连接开始发送大文件，并根据实时吞吐和阻塞情况在离散档位 `1 / 2 / 4 / 8` 之间调整并发连接数。系统 MUST 不得将固定并发度写死为单一值。

#### Scenario: 吞吐持续提升时升档
- **WHEN** 当前观察窗口内吞吐持续增长且未出现明显发送或落盘饱和
- **THEN** 系统必须将会话并发提升到下一离散档位

#### Scenario: 升档收益不足时降档
- **WHEN** 升档后吞吐未提升或波动显著恶化
- **THEN** 系统必须将会话并发降回上一个稳定档位

### Requirement: 系统必须保证接收文件的完整性与原子提交

接收端 MUST 在准备阶段创建临时文件并按目标大小预分配。接收端 MUST 按数据帧声明的 `offset` 写入目标区域，记录分片覆盖情况，并在全部覆盖完成后执行最终完整性校验。只有完整性校验通过时，系统才可以将临时文件原子提交为正式接收文件。

#### Scenario: 完整接收后提交文件
- **WHEN** 全部分片均已写入、覆盖位图完整且最终摘要校验通过
- **THEN** 系统必须将临时文件提交到正式接收目录，并将传输状态标记为完成

#### Scenario: 完整性失败时不提交文件
- **WHEN** 数据帧校验失败、覆盖不完整或最终摘要校验失败
- **THEN** 系统不得提交临时文件为正式文件，并必须将传输状态标记为失败

### Requirement: 系统必须在高速路径失败时自动回退且保持单记录语义

当极速传输无法完成时，系统 MUST 在可恢复的失败场景下自动回退到现有普通文件传输路径。回退后系统 MUST 保持原有 `transferId` 和 `messageId`，不得在用户界面中生成重复传输记录。

#### Scenario: 准备阶段失败触发回退
- **WHEN** 高速会话在能力协商、端口连接或会话准备阶段失败，且普通文件路径仍可用
- **THEN** 系统必须自动切换到普通文件传输路径，并沿用原有 `transferId` 和 `messageId`

#### Scenario: 不可恢复失败直接终止
- **WHEN** 本地文件失效、接收端磁盘不可写、接收端明确拒收或最终摘要校验失败
- **THEN** 系统必须终止该传输并标记失败，不得进行高速回退重试
