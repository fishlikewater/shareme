## Context

本次桌面化改造启动前，Message Share 的运行时由单个 Go 后台进程承载，启动时同时拉起三套职责：

- 局域网对等通信面：发现、配对、TLS 对等 HTTP、文件与极速传输。
- 本地数据面：SQLite、设备身份、下载目录与传输状态。
- 本地 UI 入口曾由 `backend/internal/api` 提供 loopback HTTP API，并由 `backend/internal/webui` embed 前端静态资源，在浏览器中访问 `http://localhost` 使用。

这套结构已经支撑了 MVP，但在本次变更启动时存在四个阻碍产品交付的问题：

- 用户入口曾是“先启动后台进程，再手动打开 localhost 页面”，不符合桌面产品直觉。
- 默认数据目录曾依赖 `UserConfigDir()/MessageShare`、`UserHomeDir()/MessageShare` 等候选路径，不满足“统一放在用户目录 `.message-share`”的产品要求。
- 本地文件选择当时仅在 Windows 通过 PowerShell + WinForms 支持，macOS 与 Linux 直接返回 unsupported。
- UI 接线、构建脚本与 embed 静态资源链路当时都围绕旧的本地 Web 模式组织，继续叠加功能会让代码边界越来越糊。

本次设计的核心不是重写传输协议，而是把现有可用的局域网传输核心放进一个跨平台桌面壳中，并把路径、构建、适配与工程结构一起拉直。

## Goals / Non-Goals

**Goals:**

- 以 Wails 作为唯一正式桌面 UI 宿主，替代手动访问 localhost 的使用方式。
- 将默认配置与运行数据根目录统一为用户主目录下的 `.message-share`，并定义清晰的目录结构。
- 让桌面 UI 在 Windows、macOS、Linux 三端都具备可运行、可构建、可分发的最小闭环。
- 保留现有局域网发现、配对、TLS 对等通信与高性能文件传输核心，不在本轮引入协议重写。
- 收敛旧入口与冗余文件，按职责重整后端宿主层、前端桥接层与构建脚本，提升可读性和维护性。

**Non-Goals:**

- 不把产品扩展到移动端、Web 远程访问、账号体系、离线消息或跨网段穿透。
- 不在本轮改写对等数据面协议为 QUIC、UDP 或其他新协议。
- 不引入复杂的设置中心、主题系统、系统托盘驻留或自动更新平台。
- 不为了“平台原生感”重新改造成三套前端；桌面 UI 仍基于现有 React 应用统一演进。

## Decisions

### 决策 1：采用“运行时核心 + Wails 桌面宿主”双层结构，移除 localhost UI 主入口

选择：

- 保留现有 `app.RuntimeService`、`protocol`、`discovery`、`transfer`、`store` 等局域网核心能力，作为独立运行时核心。
- 新增桌面宿主层，负责 Wails 生命周期、窗口管理、原生文件选择、前端事件推送与桌面命令绑定。
- 现有 `backend/internal/api` 中仅服务本地 Web UI 的 HTTP 路由不再作为正式运行路径；桌面前端直接通过 Wails bridge 调用宿主层导出的命令。
- 对等通信使用的 TLS HTTP server 继续保留，因为它属于设备间协议面，而非本地 UI 面。

原因：

- 现有 localhost UI 只是本地展示通道，不应继续占据主入口地位；桌面宿主替换它后，用户体验和分发方式才能真正成立。
- 直接桥接 `RuntimeService` 能去掉本地 HTTP/WS/SSE 这一层序列化与鉴权样板，减少维护面。
- 保留局域网协议面可最大化复用既有传输、配对和发现能力，避免桌面化把核心传输也一并打散重做。

备选方案：

- 保留本地 HTTP API，并在 Wails WebView 中继续访问 localhost：迁移最省事，但会长期保留双入口和额外通信层，不符合“清理冗余”的目标。
- 彻底重写为前后端全新桌面架构：边界更干净，但会放大范围并破坏当前已验证的核心传输实现。

### 决策 2：跨平台统一使用 `~/.message-share` 作为默认根目录，而不是按平台拆成多套系统目录

选择：

- 默认根目录统一解析为当前用户主目录下的 `.message-share`。
- 该目录作为桌面运行时唯一默认根，内部至少包括：
  - `config.json`：用户可编辑运行配置，如设备名称、显式下载目录与必要的高级覆盖项。
  - `local-device.json`：本机设备身份、公钥与设备标识。
  - `message-share.db`：SQLite 存储。
  - `downloads/`：系统下载目录不可用时的回退交付目录。
  - `logs/`：桌面运行日志。
  - `tmp/`：迁移或桌面宿主临时文件。
- 仍保留环境变量覆盖能力，但将其定位为开发、测试和高级运维手段，而不是普通用户的主要配置方式。

原因：

- 用户已经明确要求配置文件默认放在用户目录 `.message-share`，且统一路径对文档、排障、迁移与跨平台支持最直接。
- 用一个稳定根目录承载配置、身份、数据库和回退下载目录，可以消除当前“有的在 roaming，有的在 home，有的在 temp”的路径分裂。
- 将“用户配置”和“身份文件”分层保存，便于后续把可编辑配置与不可随意改动的身份材料分开维护。

备选方案：

- 继续使用 `AppData` / `~/Library/Application Support` / `~/.config` 等平台原生目录：更符合各平台传统，但与明确产品要求冲突，也会增加支持文档成本。
- 只修改配置文件位置，不统一数据库与身份目录：改动小，但仍会留下多根目录并增加排障难度。

### 决策 3：本地文件选择迁移到 Wails 原生对话框，继续保留 `LocalFileLease` 安全语义

选择：

- 废弃当前基于 PowerShell/WinForms 的 Windows 专用 picker，实现通过 Wails runtime 调用原生打开文件对话框。
- 前端请求“选择文件”时，不再经过 loopback HTTP `/api/local-files/pick`，而是调用桌面 bridge 方法。
- 桌面宿主层拿到真实路径后，仍交给 `localfile.Manager` 生成 `localFileId` 和 `LocalFileLease`，前端仅收到安全引用、展示名、大小和极速资格，不直接拿到本地路径。
- 对 macOS 与 Linux，不再走 unsupported 占位实现，而是使用同一套 Wails 原生能力承载文件选择。

原因：

- 这一步能同时解决两件事：去掉本地 HTTP UI 依赖，以及补齐 macOS/Linux 的本地文件选择缺口。
- `LocalFileLease` 已经是一个合理的边界，应该保留；改变的是“如何拿到文件”，不是“如何安全引用文件”。
- 统一通过 Wails runtime 处理原生对话框，比继续维护多套 PowerShell / osascript / zenity 差异实现更稳。

备选方案：

- 继续保留现有 `Picker` 接口并为三端分别补系统命令实现：可行，但平台分支多、维护性差。
- 允许前端直接拿到文件绝对路径：接线更短，但会破坏当前本地文件安全引用模型。

### 决策 4：使用“同一 React 应用 + 双客户端抽象”迁移前端，优先保留页面与状态模型

选择：

- 保留现有 React + Vite 前端，不重写页面结构与核心组件。
- 将当前 `frontend/src/lib/api.ts` 抽象为统一客户端接口，并以 Wails bridge 适配器作为唯一正式实现；历史 localhost 客户端不再保留在活跃生产或测试入口中。
- 事件推送从当前本地 WebSocket/SSE 模型切换为 Wails runtime event，前端仍消费统一事件对象，尽量不改消息、传输和配对状态装配逻辑。
- 文件发送分为两条路径：
  - 普通文件：桌面宿主通过原生路径直接流式读取并交给运行时，不再经过“浏览器上传到本机桌面宿主再落临时文件”的重复落盘。
  - 极速文件：继续基于 `localFileId` 引用走既有高速发送流程。

原因：

- 页面与状态模型已经在当前版本被多轮验证，桌面化阶段应优先换宿主，不应顺手重写全部前端。
- “统一客户端接口”可以把 UI 与宿主通信解耦，避免未来测试完全绑定 Wails 环境。
- 去掉“浏览器先上传到本机桌面宿主临时文件”这一跳，能直接兑现用户对本机大文件发送效率的要求。

备选方案：

- 前端整体按 Wails 重新写一套：可做得更桌面原生，但成本高且会放大回归面。
- 完全保留 HTTP 客户端：最稳，但仍会把桌面 UI 建在 localhost 之上，达不到清理目标。

### 决策 5：采用“显式迁移 + 不强删旧数据”的目录迁移策略

选择：

- 启动时先解析新根目录 `~/.message-share`。
- 若新目录不存在，而检测到旧默认目录（如 `UserConfigDir()/MessageShare` 或 `~/MessageShare`）中存在可迁移文件，则执行一次性迁移。
- 迁移对象至少包括：配置文件、`local-device.json`、SQLite 数据库、日志目录和回退下载目录元数据；大体量文件仅在需要时迁移，避免首启阻塞过久。
- 迁移成功后写入迁移标记；旧目录默认不立即删除，只记录为 legacy 源，便于回退或手工核对。
- 若新旧目录同时存在，以新目录为准，避免“自动覆盖用户已在新目录中调整过的配置”。

原因：

- 更换默认目录属于 **BREAKING** 行为，没有迁移会让用户的配对关系、设备身份或消息记录看起来像“丢了”。
- 不强删旧目录更安全，也符合桌面工具升级时“先保证用户数据不丢”的原则。
- 新旧目录并存时明确优先级，可避免再次出现“用户改了配置却被重写”的体验问题。

备选方案：

- 不做迁移，直接从新目录冷启动：实现最简单，但会破坏升级体验。
- 首启自动剪切全部旧文件并删除旧目录：路径最干净，但失败面更大，回退困难。

### 决策 6：以 Wails 工具链为主线重整构建与冗余文件清理

选择：

- 桌面开发、构建与打包统一围绕 Wails 命令与配置组织，前端由 Wails 接入现有 Vite 项目。
- 保留最小必要的脚本包装层，用于 Windows PowerShell 与 macOS/Linux shell 下的开发、测试和构建快捷入口。
- 本次变更已清理或下线以下旧链路：
  - `backend/internal/webui` embed 静态资源目录与相关接线。
  - `backend/cmd/message-share-agent` headless 主入口与相关独立启动路径。
  - 仅为 localhost UI 服务的 `dev-web.ps1`、`build-agent.ps1` 等旧脚本。
  - 生产路径上依赖 `MESSAGE_SHARE_LOCAL_API_ADDR` 与前端 `VITE_MESSAGE_SHARE_LOCAL_API_*` 的入口配置。
- 对等网络协议、测试夹具和需要 HTTP server 的设备间接口保留，不因 UI 宿主变化而删除。

原因：

- Wails 已经自带桌面构建主线，继续保留旧的“手搓 Go build + Vite build + embed”流程只会制造两套事实标准。
- 清理冗余文件是这次需求的一部分，不应把旧入口永久以“兼容”名义留在生产代码中。
- 构建入口收敛之后，Windows/macOS/Linux 的交付、测试和文档才会真正统一。

备选方案：

- 双轨长期并存：兼容性最高，但复杂度长期失控。
- 一次性清空全部旧脚本：过于激进，不利于平滑迁移测试流程。

## Risks / Trade-offs

- [Wails bridge 替换本地 HTTP 后，前端测试基建需要调整] → 保留统一客户端接口，允许测试继续注入 mock 客户端，不把组件测试绑死到 Wails 运行时。
- [统一使用 `~/.message-share` 在 Windows 上不如 AppData“原生”] → 以明确产品要求为准，并保留环境变量覆盖与迁移日志，降低运维成本。
- [首启迁移可能拖慢启动] → 采用显式迁移标记和按需迁移策略，大文件优先避免阻塞窗口首开。
- [删除 localhost UI 主入口会影响现有脚本与调试方式] → 不再保留旧 localhost 客户端实现；若未来确需协议层夹具，应以显式测试工具独立承载，而不是继续挂在当前桌面前端主线上。
- [Wails 引入新的桌面依赖与三端打包复杂度] → 将平台差异集中在宿主层与构建脚本，运行时核心保持平台无关。
- [普通文件发送去掉浏览器上传中转后，会改变部分前端接口语义] → 通过统一客户端接口屏蔽差异，前端页面仍面向“选择文件并发送”的同一交互模型。

## Migration Plan

1. 先拆分运行时核心与桌面宿主边界，确保局域网通信、存储与事件模型可被 Wails 宿主复用。
2. 引入新的目录解析与 `~/.message-share` 布局，并实现 legacy 默认目录探测与一次性迁移。
3. 用 Wails bridge 替换本地 HTTP UI 主入口，迁移文件选择、启动 Bootstrap、命令调用和事件订阅。
4. 将普通文件发送改为桌面宿主直读本地文件，消除“前端上传到本机桌面宿主临时文件”的重复落盘。
5. 接入 Windows、macOS、Linux 的构建与启动脚本，完成三端 smoke 流程。
6. 清理旧 embed Web UI、旧入口脚本与不再使用的前端环境变量接线。
7. 若桌面化上线后发现严重回归，可短期回退到 legacy 分支发版；同一二进制内不保留新的正式双入口模式。

## Open Questions

- 本轮无阻塞性开放问题。
- 若实现阶段发现某一平台的 Wails 运行前置依赖需要额外安装，以“补齐构建与安装文档”为处理方式，不扩大本次功能范围。
