## Context

当前 Message Share 已具备局域网点对点文本与文件发送能力，并已经引入了极速大文件传输与本地文件桥接。但从现网问题看，系统仍存在三个影响交付的问题：

- 极速传输的发送端按网络写入完成推进进度，而不是按接收端真正落盘完成推进，接收端慢时可能在完成阶段暴露“未写完/未覆盖完”的失败。
- 接收文件的正式落点仍偏向应用内部目录，不符合桌面用户“文件应该直接出现在下载目录”的直觉。
- Bootstrap 会把每个会话的全部消息一次性装入前端，历史增长后会拖慢首屏、切换会话与滚动体验。

本次设计要在不改动账号、跨网段与离线能力的前提下，补齐接收闭环、文件交付落点与长会话加载模型。

## Goals / Non-Goals

**Goals:**

- 让极速文件传输以“接收端确认落盘”为节拍推进，避免发送快、接收慢时传输失败。
- 让所有接收完成的文件默认进入当前用户的系统下载目录，并在目录不可用时有明确回退策略。
- 让聊天页首屏只加载最近 10 条消息，更早消息通过滚动分页按需加载。
- 保持现有事件流、消息模型与传输记录语义，避免引入第二套 UI 状态机。

**Non-Goals:**

- 不在本次设计中引入断点续传、目录传输、多文件调度或群聊历史模型。
- 不改变实时事件推送机制；新增的只是历史消息查询能力。
- 不把接收目录选择扩展为复杂的用户自定义目录管理中心；本轮只定义默认下载目录与回退规则。
- 不重写极速传输为 UDP/QUIC 等新协议；仍基于当前 TCP 数据面演进。

## Decisions

### 决策 1：极速传输改为“接收端提交确认驱动”的有界流水线

选择：

- 接收端在 `prepare` 阶段向发送端声明可接受的 `maxInFlightBytes`、推荐 `chunkSize` 与提交确认超时。
- 发送端不再仅以 socket 写入完成作为“该块已完成”，而是必须等待接收端对该块的落盘确认后，才能释放窗口继续发送后续块。
- 完成阶段必须以“全部块已收到接收端确认 + 覆盖完整 + 最终摘要一致”为前提，才能发送 complete。

原因：

- 当前 TCP 自带的背压只能保证对端内核缓冲区被消费，不能保证应用层已经 `WriteAt` 成功。
- 发送端只有拿到接收端提交确认，才能避免 complete 抢在接收端落盘完成之前发生。
- 有界窗口能在接收磁盘慢、CPU 忙或机械盘抖动时主动收敛，而不是让发送端持续把数据推到不可控状态。

备选方案：

- 继续依赖 TCP 背压：实现最小，但不能解决“写入完成”和“落盘完成”语义不一致的问题。
- 退回单连接串行写入：稳定但会直接牺牲局域网吞吐上限，不符合产品目标。

### 决策 2：接收文件采用“统一最终交付目录解析器”

选择：

- 启动时解析默认接收目录，优先级为：显式环境变量配置 > 系统下载目录 > 现有 `DataDir/downloads` 回退目录。
- 普通文件接收与极速文件接收共用同一套最终提交策略，统一处理目录创建、文件名净化、重名避让与原子提交。
- UI 与日志只展示最终文件名与最终目录策略结果，不暴露内部临时文件路径。

原因：

- 用户对桌面文件传输工具的默认预期就是“收到的文件进下载目录”，这比应用私有目录更可用。
- 普通路径与极速路径若分裂成两套目录策略，会导致用户在不同发送方式下得到不同结果，体验不一致。
- 保留环境变量覆盖与回退目录，可避免在系统目录不可解析、权限受限时直接失效。

备选方案：

- 继续使用应用数据目录：实现简单，但不符合桌面用户习惯。
- 引入可配置目录设置页：更灵活，但超出本次问题修复范围。

### 决策 3：Bootstrap 仅承担“最近 10 条消息窗口”，历史改为显式分页接口

选择：

- `Bootstrap` 对每个会话最多只返回最近 10 条消息，仍保留会话列表、最近一条预览与传输态汇总。
- 新增本地 API 历史查询接口，按会话和游标向前分页加载更早消息；每页固定 10 条。
- 前端在聊天面板向上滚动接近顶部时触发下一页请求，并把旧消息前插到当前列表中。

原因：

- 现有 Bootstrap 把全部消息装入内存，随着使用时间增长会线性放大首屏负担。
- 把“首屏窗口”和“历史翻页”分离后，既能保持进入会话的即时感，也能让长会话继续增长而不拖垮 UI。
- 固定页大小和向前分页最符合聊天界面的浏览模型，复杂度也低于双向任意窗口。

备选方案：

- 完全不在 Bootstrap 返回消息：首屏会话体验变慢，且需要额外补最近一条预览接口。
- 继续一次性返回全部消息：实现最简单，但已经证明会带来体验与性能问题。

### 决策 4：分页游标采用“时间戳 + messageId”的稳定游标，而不是页码

选择：

- 历史分页接口返回一个由最旧消息生成的下一页游标，内部语义为 `createdAt + messageId` 的稳定边界。
- 数据层按 `created_at desc, message_id desc` 检索，再在返回前转回时间正序，保证聊天显示顺序不变。
- 为消息表增加面向 `conversation_id + created_at + message_id` 的索引，避免分页查询退化。

原因：

- 聊天记录是持续插入的数据集，页码会在新消息到来时发生漂移。
- 单独依赖时间戳在极端情况下会遇到同一时刻多条消息并列的问题，附带 `messageId` 可消除歧义。
- 数据库层按降序取最近 N 条，再转正序返回，既贴合索引，也容易维护 UI 顺序。

备选方案：

- 页码分页：实现直观，但在实时消息流下不稳定。
- 只用 `beforeCreatedAt`：实现更简单，但边界去重不够稳。

## Risks / Trade-offs

- [接收确认让数据面协议更复杂] → 通过“每块确认 + 有界窗口”控制在最小可用闭环，不扩展成通用流式重传协议。
- [系统下载目录在不同 Windows 环境下解析方式不同] → 优先使用系统目录解析，失败时明确回退到 `DataDir/downloads`，保证功能不阻塞。
- [Bootstrap 改小后前端状态装配方式需要调整] → 保持实时事件追加逻辑不变，只把历史查询职责移到单独接口和滚动触发器。
- [历史分页查询增加数据库负担] → 增加组合索引，并固定页大小为 10，避免无界扫描。
- [普通文件与极速文件统一目录策略会触及更多接收路径] → 统一通过公共文件落点组件收敛，减少分支散落。

## Migration Plan

1. 先引入统一接收目录解析器，并让普通/极速接收都改走该组件。
2. 为极速传输补接收确认与发送窗口控制，先保证失败条件和完成语义收紧，再调 UI 文案。
3. 增加消息分页查询接口、数据访问索引和 Bootstrap 最近 10 条窗口逻辑。
4. 前端切换为“首屏窗口 + 向上滚动分页加载”的消息装配方式，并保留现有实时事件追加逻辑。
5. 若上线后发现分页或目录策略问题，可单独回退 Bootstrap 裁剪或目录解析逻辑，不影响既有消息存储结构。

## Open Questions

- 本轮无阻塞性开放问题；若实现时发现系统下载目录解析在特定 Windows 版本存在兼容差异，以“系统目录失败即回退应用目录”为准，不再扩大需求范围。
