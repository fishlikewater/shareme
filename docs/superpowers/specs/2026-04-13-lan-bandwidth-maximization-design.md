# 局域网带宽榨取与大文件并行分片设计

## 背景

当前项目已经具备以下能力：

- 基于 UDP 广播的局域网设备发现
- 基于 TLS 的 agent 间点对点消息与文件传输
- 本地 Web UI 与单 exe 交付形态
- 基础传输 telemetry、速率与进度展示

最近一轮优化已经解决了以下问题：

- 已配对设备的可达性通过 heartbeat 自动恢复
- 文件传输链路改为流式 multipart
- 单流链路已经具备 256 KiB 拷贝缓冲
- 中间 `transfer.updated` 事件已做节流

但如果目标升级为“尽量榨干局域网带宽”，当前仍然存在两个核心限制：

1. 数据面仍以单连接单流传输为主，大文件吞吐上限容易受单连接 TCP/TLS 性能限制。
2. 本地 Web UI 上传入口仍然会先把浏览器上传内容落到 agent 本地临时文件，再由 agent 重新读取并发送给对端，这会把链路变成“浏览器 -> 本地磁盘 -> 对端”，不利于跑满带宽。

本次设计目标是在不引入过高系统复杂度的前提下，找到“复杂度 / 性能 / 稳定性”的最佳平衡点。

## 目标

### 产品目标

- 在局域网环境下显著提升大文件传输峰值吞吐。
- 保持现有“单 exe + 本地 Web UI + agent 对 agent”的产品形态不变。
- 对小文件和文字消息不引入不必要复杂度。
- 保持现有 TLS 信任模型、配对模型与可达性模型。
- 让后续支持群聊、广播、断点续传时的改动成本尽量降低。

### 技术目标

- 明确数据面协议选型，避免过早切入复杂度过高的协议。
- 小文件走低开销单流路径，大文件走并行分片路径。
- 并行分片具备可重试、可校验、可汇总进度的能力。
- 当前 UI 仍只暴露“单个文件传输任务”，不泄漏内部 chunk 细节。

## 非目标

- 跨网段穿透
- 离线消息
- 账号体系
- 浏览器直连浏览器
- 本轮内的断点续传
- 本轮内的 QUIC / HTTP/3 数据面切换
- 本轮内的群聊和公告板广播

## 方案比较

### 方案 1：继续单流，仅做链路极限优化

做法：

- 去掉本地 Web 上传的临时文件中转
- 显式启用 HTTP/2
- 调优 `http.Transport`
- 继续扩大读写缓冲与复用连接

优点：

- 复杂度最低
- 改造面相对最小
- 对现有状态模型影响小

缺点：

- 大文件吞吐提升有限
- 很难真正突破单连接的性能上限
- 对“榨干局域网带宽”目标支撑不足

### 方案 2：HTTP/2 + 大文件并行分片，同时带上单流链路优化

做法：

- 小文件保持单流路径
- 大文件先创建传输会话，再并行上传多个分片
- 接收端按 offset 写入同一个临时文件
- 全部分片完成后做完整性校验，再原子提交
- 同时对单流路径做“去磁盘中转 + 启用 HTTP/2 + transport 调优”

优点：

- 复杂度、性能、稳定性之间平衡最好
- 最有机会显著逼近局域网带宽上限
- 继续复用现有 TLS、HTTP、配对与事件模型
- 后续演进到断点续传、广播任务时边界清晰

缺点：

- 需要新增分片会话状态与协调逻辑
- 实现与测试复杂度中等

### 方案 3：切换到 QUIC / HTTP/3

做法：

- 数据面整体迁移为 QUIC / HTTP/3
- 利用多路流和更现代的传输控制能力提升吞吐

优点：

- 长期演进空间较好
- 在复杂网络环境中具备潜力

缺点：

- 当前项目是局域网点对点，不一定比调优后的 TCP/TLS 获得更大收益
- 引入 UDP 数据面、额外 transport 栈与更多排障复杂度
- 不适合作为当前 V1 的第一优先级

## 选型结论

采用方案 2：

- 数据面继续使用 `HTTP/2 + TLS/TCP`
- 小文件继续走单流，但去掉本地磁盘中转并调优 transport
- 大文件启用并行分片会话

原因：

1. 对当前产品目标最契合，能实实在在提升大文件峰值吞吐。
2. 不需要推翻现有 agent 架构、信任模型和前端模型。
3. 比直接上 QUIC 更容易稳定落地，也更容易测试和回归。
4. 后续如需支持 QUIC，可在 transport 抽象上平滑扩展。

## 协议与数据流设计

### 保留现有协议分层

- UDP：仅用于设备发现与广播公告
- HTTP/2 over TLS/TCP：用于配对、消息、heartbeat、文件传输

### 传输路径选择

- 小文件：`fileSize < 64 MB`
  - 继续走现有单文件上传接口
  - 使用直通流式路径
- 大文件：`fileSize >= 64 MB`
  - 进入“传输会话 + 并行分片”路径

### 大文件并行分片协议

新增三类 peer 接口：

1. `POST /peer/transfers/session/start`
2. `POST /peer/transfers/session/part`
3. `POST /peer/transfers/session/complete`

#### 1. 创建传输会话

发送端先向接收端申请会话，输入：

- `transferId`
- `messageId`
- `senderDeviceId`
- `fileName`
- `fileSize`
- `fileSha256`
- `agentTcpPort`

接收端返回：

- `sessionId`
- `chunkSize`
- `parallelism`
- `expiresAt`

作用：

- 接收端预创建临时文件和会话状态
- 双方协商 chunk 参数
- 把本次大文件传输映射为一个稳定 session

#### 2. 上传分片

每个分片独立请求，元信息包含：

- `sessionId`
- `transferId`
- `partIndex`
- `offset`
- `length`
- 可选 `partSha256`

body 为该分片的原始字节流。

接收端逻辑：

- 校验 session 是否有效
- 通过 `offset` 写入同一个临时文件
- 记录分片完成状态
- 返回分片确认结果

#### 3. 完成提交

当所有分片成功后，发送端发 `complete` 请求，包含：

- `sessionId`
- `transferId`
- `totalSize`
- `fileSha256`
- `partCount`

接收端完成：

- 校验所有分片是否齐全
- 校验总大小
- 校验整文件 sha256
- 原子提交临时文件
- 写入最终 transfer outcome

## 参数策略

### 默认参数

- 并行阈值：`64 MB`
- 分片大小：`8 MB`
- 默认并发数：`4`
- 最大并发数：`8`
- 单分片重试次数：`2`
- 整文件校验：默认 `sha256`

### 参数选择原因

- `8 MB` 在请求开销与失败重传成本之间较平衡
- `4` 路并发通常足以突破单连接瓶颈
- `8` 路作为上限，可应对更高带宽或更快磁盘
- 整文件校验能为并行随机写入提供最终一致性保障

### 自适应策略

V1 中默认使用固定并发 `4`。实现上预留可扩展点，后续可根据实测速率调整并发到 `6` 或 `8`，但本轮不做动态调参算法。

## 单流路径优化

即使大文件走并行分片，单流路径仍需继续优化。

### 1. 去掉本地 Web 上传临时文件中转

当前本地 API 上传入口会先把浏览器上传内容写入本地临时文件，再读取临时文件交给 `app.SendFile`。

本轮应改为：

- 浏览器上传流直接进入发送链路
- 对单流路径不再做完整本地落盘中转

这样可减少：

- 一次完整磁盘写入
- 一次完整磁盘读取
- 本地磁盘成为瓶颈的概率

### 2. 显式启用 HTTP/2

当前项目使用自定义 `TLSClientConfig`，应显式确认数据面 client 启用 HTTP/2，避免落回仅支持 HTTP/1.1 的保守配置。

### 3. 调优 `http.Transport`

推荐调优项：

- `ForceAttemptHTTP2 = true`
- `MaxIdleConns`
- `MaxIdleConnsPerHost`
- `MaxConnsPerHost`
- `IdleConnTimeout`
- `WriteBufferSize`
- `ReadBufferSize`
- 关闭不必要的默认限制

目标：

- 为多连接并行上传提供足够的连接池空间
- 避免默认 transport 配置限制大文件吞吐

## 模块设计

### 1. `app/service.go`

职责：

- 判断走单流还是大文件分片
- 创建统一的用户可见 `transferId`
- 汇总分片进度为一个传输任务
- 发布对前端可见的 `transfer.updated`

要求：

- UI 仍然只看到单个传输任务
- chunk 细节隐藏在 agent 内部

### 2. 新增传输协调器

建议新增 `backend/internal/transfer/session_*`：

- `session_coordinator.go`
- `session_sender.go`
- `session_receiver.go`
- `session_store.go`

职责：

- 创建与管理 session
- 切分 chunk
- 调度 worker 并发发送
- 处理 chunk 重试
- 汇总进度

### 3. `protocol/peer_http.go`

职责：

- 新增会话型分片接口
- 保持现有 TLS 鉴权模式
- 保持 request authorizer 结构一致

### 4. 接收端文件写入器

建议新增“按 offset 写入临时文件”的 writer。

要求：

- 能随机写
- 能记录已完成 chunk
- 能在 `complete` 前保持临时态
- 能在失败时清理

### 5. Store 层

本轮只保存最终用户可见状态：

- transfer 主记录
- 消息记录

chunk 级与 session 级状态先保留在内存中，不做持久化。

原因：

- 本轮不做断点续传
- 驻内存实现更轻
- 可以明显降低 V1 复杂度

## 进度与事件模型

### 统一对外传输视图

无论单流还是并行分片，前端都只看到一个 `transfer.updated`：

- `BytesTransferred`
- `ProgressPercent`
- `RateBytesPerSec`
- `EtaSeconds`
- `State`

### 分片进度聚合

每个 chunk worker 仅更新内部 session telemetry。

外部事件仍经过统一节流门：

- `minPublishInterval = 120 ms`
- `minPublishBytes = 256 KiB`
- 终态事件立即发布

这样可以避免：

- chunk 并发带来的事件风暴
- UI 高频抖动
- 速率计算噪声放大

## 稳定性与失败处理

### 分片失败

- 单片失败仅重试当前 chunk
- 超过重试阈值则整文件转 `failed`

### 文件提交

- 所有 chunk 成功后才允许 `complete`
- `complete` 校验失败则整体失败，不提交正式文件

### 清理策略

- 失败 session 清理临时文件
- 过期 session 定时清理

## 演进兼容性

本方案为后续变化轴预留了扩展空间：

### 1. 群聊 / 广播

未来可在发送端复用同一个分片调度器，把同一个文件推送到多个 peer，但每个 peer 仍保持独立 session。

### 2. 断点续传

未来只需把当前内存中的 chunk 完成状态和 session 状态落盘，即可演进到续传。

### 3. QUIC / HTTP/3

未来可新增一个 QUIC transport 实现，并复用当前的 session 协议语义，而不需要推翻传输协调器层。

## 验收标准

满足以下条件视为本轮设计达标：

1. 小文件路径仍保持稳定，且由于去本地磁盘中转而有可见收益。
2. 大文件在局域网环境下吞吐显著高于当前单流路径。
3. 大文件传输失败时仅重试 chunk，不产生半截正式文件。
4. 前端仍只展示一个传输任务，不暴露 chunk 细节。
5. 现有 heartbeat、文字消息、小文件传输不受回归影响。
6. 后端、前端、脚本测试与打包链路仍可通过。

## 风险

- 并发过高会把瓶颈从网络转移到磁盘或 CPU，因此默认并发不应直接拉满。
- 若使用机械硬盘，随机写性能可能成为接收端瓶颈，需要通过顺序预分配和合理 chunk 大小缓解。
- 去掉本地 Web 上传临时文件中转后，接口和状态流需要重新梳理，否则容易把现有 `SendFile` 签名挤得过重。
- 本轮不做 session 持久化，因此 agent 进程重启后大文件任务不恢复；这属于非目标内可接受范围。

## 结论

本轮最优平衡方案为：

- UDP 仅做发现
- 数据面继续使用 `HTTP/2 + TLS/TCP`
- 小文件走单流直通优化
- 大文件启用并行分片会话
- 本轮不切换到 QUIC / HTTP/3

这是当前阶段在复杂度、性能、稳定性之间最适合落地的方案。
