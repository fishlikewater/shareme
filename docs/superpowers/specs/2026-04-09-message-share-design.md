# 局域网消息分享工具设计文档

## 1. 背景

目标产品是一款运行在局域网环境中的消息与文件分享工具。每台电脑运行一个轻量本地代理进程，用户通过浏览器访问本机 Web 页面完成文字消息和文件发送。

本项目的核心约束如下：

- 无固定服务端
- 每台电脑均可对等通信
- 第一版仅支持 Windows
- 自动发现为主，手动输入 IP 为兜底
- 第一版以点对点发送为核心
- 产品形态为聊天式会话，而非一次性投递
- 仅支持双方在线时的实时发送
- 会话历史长期保存在本机
- 已配对设备默认自动接收
- 第一优先级是简单好用
- 第一版支持边界限定为同一二层或同一子网、未开启客户端隔离、允许入站监听端口的家庭或小型办公局域网

## 2. 设计目标

本设计旨在实现以下目标：

- 在同一局域网内自动发现其他在线设备
- 支持双方确认后建立信任关系
- 支持已信任设备之间发送文字消息与文件
- 支持本机长期保存聊天历史和文件记录
- 通过浏览器页面提供低学习成本的交互体验
- 在架构上为后续公告板广播、群发和跨平台扩展预留空间

## 3. 非目标

第一版不包含以下能力：

- 群聊
- 公告板广播
- 离线消息
- 中心节点中转
- 断点续传
- 跨网段穿透
- 多端历史同步
- 账号体系
- 富文本消息

## 4. 总体架构

系统采用“本地代理 + Web UI + 设备间点对点通信”的分层架构。

### 4.1 架构原则

- 浏览器只负责界面，不直接参与局域网发现和设备间协议
- 本地代理负责设备发现、配对、通信、存储和系统集成
- 设备之间通过点对点连接交换消息与文件
- 所有本地状态以代理进程为唯一真相源
- 本地 API 仅监听 `127.0.0.1`
- 所有本地写操作接口必须做会话令牌校验与 `Origin` 校验，避免任意网页跨站调用本机代理

### 4.2 逻辑分层

#### 1）Web UI 层

负责以下能力：

- 设备列表展示
- 会话消息展示
- 文本输入与文件拖拽发送
- 发送进度与异常提示
- 基础设置管理

#### 2）本地 API 层

浏览器通过本机 `HTTP + WebSocket` 与代理通信：

- HTTP 用于命令型操作
- WebSocket 用于推送在线状态、消息、进度和系统事件
- 前端首次进入页面时先拉取状态快照，再订阅增量事件

#### 3）发现与配对层

负责以下职责：

- 自动发现局域网在线设备
- 手动输入 IP 连接
- 发起和接收配对请求
- 保存信任关系

#### 4）会话与传输层

负责以下职责：

- 文本消息发送和接收
- 文件元数据交换
- 文件分块传输
- 传输状态管理

#### 5）本地存储层

负责持久化以下数据：

- 本机设备身份
- 本机设备配置
- 已信任设备信息
- 聊天会话
- 消息历史
- 文件索引

#### 6）系统集成层

预留以下系统能力：

- 托盘
- 开机自启
- Windows 通知
- 防火墙提示
- 默认下载目录管理

## 5. 核心模块设计

### 5.1 设备身份模块

首次启动时生成并持久化本机身份信息：

- `device_id`
- `device_name`
- 长期公私钥对
- 本地监听端口
- 版本号和能力描述

其中 `device_id` 是设备的唯一稳定标识，不能使用 IP 代替。IP 仅作为临时网络地址。

### 5.2 设备名称配置规则

这是本次设计新增确认的约束。

#### 规则

- 代理端必须支持用户自定义本机设备名称
- 设备名称属于本机配置项，需持久化保存
- 自动发现广播必须携带该配置名称
- 其他设备发现后，在设备列表中默认显示该配置名称

#### 补充显示规则

- 内部识别一律使用 `device_id`
- UI 默认展示 `device_name`
- 如果出现重名设备，可附加备注名、指纹缩写或其他区分信息
- 后续可支持用户为远端设备设置本地备注名，但不改变远端真实配置名称

### 5.3 设备发现模块

发现模块负责找到当前局域网内在线设备。

#### 发现方式

- 主方式：同一子网内使用 `UDP 广播`
- 兜底方式：手动输入 `IP:port`
- 第一版不把 `mDNS` 作为必选依赖，避免增加 Windows 侧环境差异

#### 支持边界

- 仅保证同一子网内自动发现
- 不保证跨 VLAN、访客网络、开启 AP Isolation 的网络可自动发现
- 如果自动发现失败，必须在 UI 中提供手动连接入口和网络诊断提示

#### 广播内容建议

```json
{
  "deviceId": "dev-123",
  "deviceName": "办公室电脑",
  "listenPort": 19090,
  "fingerprint": "ab:cd:ef",
  "version": "0.1.0",
  "capabilities": ["text", "file"]
}
```

#### 设计原则

- 发现层只解决“谁在线”
- 不在发现层传输消息正文和文件内容
- 设备在线表通过心跳或广播时间做过期清理
- 自动发现属于不可靠信道，发现成功不等于业务连接一定可达

### 5.4 配对与信任模块

配对模块负责建立“后续可自动接收”的信任基础。

#### 首次配对流程

1. A 发现 B
2. A 发起配对请求
3. 双方建立临时握手并各自展示同一组 6 位短码
4. B 收到请求并核对短码后确认
5. A 也核对短码并确认
6. 双方都确认后建立信任关系
7. 本地持久化保存对方身份信息和公钥指纹

#### 信任信息建议

- 对方 `device_id`
- 对方 `device_name`
- 对方公钥或指纹
- 对方 TLS 证书或公钥 pin 信息
- 首次配对时间
- 信任状态
- 本地备注名

### 5.5 会话与消息模块

每个已信任设备对应一个聊天会话。

#### 第一版支持的消息类型

- `text`
- `file`
- `system`

#### system 消息示例

- 已建立信任关系
- 对方上线
- 对方离线
- 文件发送完成
- 文件发送失败

#### 会话规则

- 会话历史长期保存在本机
- 对方离线时仍可查看历史
- 第一版不支持离线待发送
- 对方离线时发送入口需明确提示不可发送

#### 接收策略

- 文本消息对已信任设备默认自动接收
- 文件对已信任设备默认自动接收到受控目录，设置页允许切换为“接收前确认”
- 默认接收目录建议为 `下载\MessageShare\<设备名称>\`
- 同名文件不覆盖，采用自动重命名
- 当磁盘空间不足、目录不可写或文件超过本机配置阈值时，必须拒收并给出明确提示

### 5.6 传输模块

传输模块分为控制通道和数据通道。

#### 连接策略

- 每台代理固定监听 TCP 端口，默认建议为 `19090`，允许在设置中修改
- 发送方在需要发消息或文件时主动拨号到对端最近一次可用的 `IP:port`
- 如果双方同时建立出站连接导致重复连接，按 `device_id` 字典序裁决，仅保留一条业务连接
- 所有控制通道和文件数据通道都运行在 TLS 之上
- 首次配对完成后，后续连接必须校验证书或公钥 pin 信息

#### 控制通道职责

- 建立连接
- 发送文本消息
- 发送文件元数据
- 回传接收状态
- 推送进度和结果

#### 数据通道职责

- 文件字节流传输
- 分块发送
- 分块接收
- 完整性校验

#### 第一版建议流程

1. 发送方发出 `file_offer`
2. 接收方基于已信任关系自动接受，回 `file_accept`
3. 发送方按分块发送文件内容
4. 接收方写入临时文件
5. 传输结束后进行哈希校验
6. 校验通过后原子改名为正式文件
7. 更新数据库中的文件状态为完成

#### 文件状态机

- `queued`
- `sending`
- `receiving`
- `verified`
- `done`
- `failed`
- `canceled`

任何文件消息都必须显式处于上述状态之一，前端显示与数据库状态必须一致。

### 5.7 本地 API 与前端状态同步模型

代理是唯一真相源，前端只持有可重建视图状态。

#### 同步规则

- 页面初始化先通过 HTTP 获取启动快照
- WebSocket 建立后从代理接收增量事件
- 每条事件必须携带递增的 `event_seq`
- 前端断线重连后，需带上最后成功处理的 `event_seq` 请求 `resync`
- 所有前端命令都必须携带 `client_request_id`，用于幂等处理

#### 目标

- 支持多标签页一致显示
- 支持断线后状态补偿
- 避免重复点击导致重复发送或状态回滚

## 6. 页面与交互设计

### 6.1 MVP 页面范围

第一版建议仅包含以下页面或视图：

- 主聊天页
- 设备发现页
- 配对确认弹层
- 设置页

首次启动引导与健康检查不单独拆分为复杂页面，可作为主聊天页空态和设置页中的引导模块存在。

### 6.2 主聊天页布局

主聊天页建议采用三段式布局：

- 左侧：设备列表
- 中间：当前会话
- 底部：发送区

#### 设备列表展示内容

- 设备名称
- 在线状态
- 是否已配对
- 最近会话
- 未读消息数

其中设备名称默认显示远端代理配置的 `device_name`。

### 6.3 关键交互

#### 发现与配对

- 用户点击“添加设备”
- 进入发现页查看在线设备
- 选择目标设备并发起配对
- 双方核对 6 位短码并确认后建立信任关系

#### 会话发送

- 选择设备进入会话
- 输入文字后发送
- 或拖拽文件进入发送区
- UI 展示发送中、完成、失败等状态

#### 离线语义

- 离线时保留历史
- 输入区提示对方离线
- 不支持离线待发

#### 首次上手闭环

1. 安装并启动本地代理
2. 代理执行本机自检，检查监听端口、防火墙、默认下载目录和本地 API 服务
3. 浏览器打开本机页面
4. 查看已发现设备，或手动输入 `IP:port`
5. 完成首次配对与短码核对
6. 发送一条测试消息或测试文件，验证主链路打通

## 7. 技术选型建议

## 7.1 候选方案

### 方案 A：Go + React + SQLite

- 本地代理：Go
- Web 前端：React + Vite
- 本地数据库：SQLite
- 本地 API：HTTP + WebSocket
- 设备发现：UDP 广播（同一子网）+ 手动 `IP:port`
- 点对点传输：TCP

### 方案 B：Rust + React + SQLite

- 更强调底层性能和长期演进能力
- 开发成本更高

### 方案 C：Electron/Tauri 一体化

- 更偏桌面壳应用
- 不完全符合“浏览器访问本机页面”的目标

## 7.2 推荐方案

推荐使用 `Go + React + SQLite + TCP`。

### 原因

- Go 适合常驻本地代理和局域网网络编程
- React 适合快速构建聊天式 Web 页面
- SQLite 适合单机本地长期保存数据
- TCP 足以满足第一版点对点实时通信要求

## 8. 协议建议

### 8.1 发现协议

用于设备在线发现，广播最小必要信息，不承载业务消息。

发现广播必须包含 `protocol_version`，并带有过期时间或心跳更新时间。

### 8.2 会话控制协议

建议支持以下消息类型：

- `pair_request`
- `pair_accept`
- `text_message`
- `file_offer`
- `file_accept`
- `file_progress`
- `file_done`
- `error`

每条控制消息都必须包含以下基础字段：

- `protocol_version`
- `request_id`
- `message_id`
- `sender_device_id`
- `error_code`（仅错误消息必填）

### 8.3 文件数据协议

建议采用固定块大小的分块传输：

- 每块带序号
- 传输完成后做整体哈希校验
- 失败后明确记录失败状态
- 必须支持取消、失败重试和重复请求幂等处理

### 8.4 本地 API 安全边界

- 本地 API 仅对 `127.0.0.1` 提供服务
- 所有写操作都需要携带代理签发的会话令牌
- 代理对 Web 请求执行 `Origin` 校验
- 不允许局域网其他设备直接访问本地代理的 HTTP 接口

## 9. 风险与异常处理

### 9.1 关键风险

- Windows 防火墙拦截
- 自动发现不稳定
- IP 变化
- 设备重名
- 已配对但连接失败
- 大文件传输中断
- 消息状态与文件状态不一致
- 代理进程异常退出

### 9.2 异常处理原则

- 错误信息必须翻译成用户可理解的语言
- 文件接收必须先写临时文件
- 所有状态以代理进程为准
- 宁可明确失败，也不要假成功

### 9.3 可诊断性要求

第一版必须提供最小化自检与诊断能力：

- 本机健康状态页
- 当前监听端口与防火墙检查结果
- 自动发现是否正常
- 最近一次连接失败原因
- 一键导出脱敏日志包

## 10. 测试策略

### 10.1 单元测试

重点覆盖：

- 身份生成
- 配对状态机
- 消息编解码
- 文件分块与重组
- 状态迁移

### 10.2 集成测试

重点覆盖：

- 发现模块与配对模块协作
- 会话模块与存储模块协作
- 文件传输与状态回写

### 10.3 端到端测试

至少覆盖以下流程：

- 发现设备
- 配对成功
- 发送文字
- 发送小文件
- 发送大文件
- 离线不可发送
- 上线后状态恢复

### 10.4 局域网真实环境冒烟测试

至少验证以下场景：

- 同一路由器下两台 Windows 设备通信
- 防火墙存在差异的机器
- 重名设备展示
- 大文件传输
- 传输中断后失败提示
- 多网卡环境（WiFi + 有线 / VPN / 虚拟网卡）
- 睡眠唤醒与网络切换
- 不同 Windows 10/11 组合
- 不同浏览器访问本机页面

### 10.5 发布前检查清单

发布前至少完成以下检查：

- 端口占用检测与冲突提示
- 防火墙规则创建、撤销和失败提示验证
- 普通用户权限下首次运行验证
- 升级、卸载和重装后的配置与历史保留策略验证
- 代理崩溃后的自动恢复与安全模式启动验证
- 失败路径验收：发现失败、连接失败、拒收、磁盘不足、目录不可写、传输中断

## 11. MVP 验收标准

满足以下条件时，第一版可视为达成目标：

1. 两台 Windows 电脑安装后可以自动发现彼此
2. 首次配对流程清晰且可完成，并可核对 6 位短码
3. 已信任设备之间可实时发送文字
4. 已信任设备之间可实时发送文件
5. 文件发送过程有明确进度和结果
6. 聊天历史在重启后仍可查看
7. 离线时界面明确提示不可发送
8. 设备列表正确显示远端设备配置名称
9. 常见异常有明确且可理解的提示
10. 本机健康检查页可定位防火墙、端口和自动发现的基础问题
11. 文件失败、取消、拒收和磁盘不足等失败路径有一致的状态与提示

- 公告板广播

## 13. 结论

本项目的推荐落地形态为：

一款运行在 Windows 上的局域网消息分享工具。每台设备运行本地代理进程，用户通过浏览器访问本机页面完成设备发现、配对、聊天和文件传输。设备之间通过局域网点对点通信，聊天历史长期保存在本机。第一版以简单好用为最高优先级，同时为后续广播、群发和跨平台演进保留清晰边界。
