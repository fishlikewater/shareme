# 局域网极速大文件传输设计文档

## 1. 文档目的

本文档定义 Message Share 在局域网场景下的“极速大文件传输”方案。目标是在不推翻现有点对点消息/文件基础能力的前提下，为单个大文件传输提供接近链路上限的吞吐能力，同时保持实现复杂度、稳定性和可落地性之间的平衡。

本文档只覆盖设计，不进入实现计划与编码。

## 2. 背景与现状

当前系统已经具备如下基础能力：

- 本地 `Web UI` 通过 `HTTP + WebSocket` 访问本机 agent。
- 点对点控制链路已经存在，设备间可完成配对、消息发送、文件传输。
- 前端当前文件发送路径仍是浏览器 `File -> FormData -> /api/transfers/file`。
- 本地 API 服务端当前使用 multipart 流式读取上传内容，并直接把流交给应用层，不存在“先落本机临时文件再二次发送”的当前实现。

当前方案在“单个大文件吃满局域网带宽”这个目标上存在天然瓶颈：

- 浏览器文件输入流不是为极限吞吐优化设计的。
- 现有路径仍然要经过浏览器到本地 agent 的一次用户态搬运。
- 现有文件发送主要沿用控制面/HTTP 路径，难以把大文件数据面做到足够激进。
- 对于局域网可信配对设备之间的大文件发送，TLS 与通用 HTTP 语义会引入额外 CPU 和协议开销。

因此，本次设计不再继续只围绕现有 Web 上传链路微调，而是为“大文件极速发送”引入专用路径。

## 3. 设计目标

### 3.1 核心目标

- 面向局域网已配对设备，显著提升单个大文件的峰值吞吐。
- 优先压榨有线局域网带宽，其次兼顾高速 Wi-Fi 场景。
- 尽量减少发送端本机的无意义 I/O 和内存拷贝。
- 保持用户侧入口简单，仍通过本机 Web UI 操作。
- 失败时自动回退到现有文件传输路径，保证功能闭环。

### 3.2 设计原则

- 控制面和数据面分离。
- 小文件继续走现有通路，大文件才进入高速模式。
- 高速模式只为局域网、已配对、单大文件优化，不为了抽象而抽象。
- V1 先保证 Windows 可交付，不把 Linux/macOS 原生桥接一起拉进本轮范围。

### 3.3 性能目标

- 在发送端和接收端均为 SSD、同交换机千兆或更高速局域网环境下，单个大文件吞吐应尽量接近 `min(链路上限, 磁盘顺序读写上限)`。
- 以 1 秒滑动窗口统计速度时，极速模式应显著高于现有 HTTP 文件路径。
- 允许因磁盘、网卡、驱动、交换机或 Wi-Fi 条件导致达不到理论极限，但架构上不能人为卡住带宽。

## 4. 非目标

本次设计明确不覆盖以下内容：

- 离线消息
- 跨网段穿透
- 多端同步
- 账号体系
- 群聊
- 公告板广播
- 目录发送
- 多文件批量极速调度
- 断点续传
- QUIC/UDP 实验分支
- Linux/macOS 原生文件桥接

## 5. 方案比较与选型结论

### 5.1 方案 A：继续优化现有 HTTP 文件传输

做法：

- 保持浏览器 `input[type=file]`。
- 继续通过本地 `multipart/form-data` 上传到 agent。
- 对现有传输会话、分片大小、缓冲区和并发度做调优。

优点：

- 改动最小。
- 与现有代码复用度高。

缺点：

- 浏览器仍是大文件数据源，发送端链路上多了一跳。
- HTTP/TLS 路径不适合极限带宽压榨。
- 吞吐上限更容易受浏览器、FormData、通用协议开销限制。

结论：

- 不适合作为“大文件极限吞吐”主方案。

### 5.2 方案 B：控制面保留 HTTPS/TLS，新增专用高速 TCP 数据面

做法：

- 控制面继续使用现有 agent 间安全控制链路。
- 仅对大文件建立专用高速数据通道。
- 数据面使用独立数据端口和专用明文 TCP。
- 采用 `TCP striping`，接收端预分配文件并按 offset 并发写入。
- Web UI 不再直接作为极速大文件的数据源，而是通过 native file bridge 让 agent 直接读取本地磁盘文件。

优点：

- 性能收益最直接。
- 复杂度显著低于自研 UDP/QUIC 可靠传输。
- 兼顾可落地性、稳定性和吞吐上限。
- 与现有控制链路兼容，失败回退容易实现。

缺点：

- 需要新增本地原生文件选择桥接。
- 需要新增数据面监听端口和会话鉴权。

结论：

- 作为本次设计的推荐方案。

### 5.3 方案 C：自研 UDP/QUIC 风格高速传输

做法：

- 使用 UDP/QUIC 或类 QUIC 自定义协议承载大文件数据面。

优点：

- 理论上可以更细粒度控制并发、拥塞和恢复策略。

缺点：

- 复杂度和调试成本显著更高。
- 在 Windows、防火墙、企业网络、部分路由器和 Wi-Fi 环境下更容易出现不确定性。
- 要把可靠性、拥塞控制、乱序、重传、流控全部重新拉进系统复杂度。

结论：

- 不适合作为当前 V1 正式交付方案。

### 5.4 最终选型

最终采用：

- 小文件：继续沿用现有浏览器上传路径。
- 大文件：采用“`现有 HTTPS/TLS 控制面 + 原生文件桥接 + 专用明文 TCP striping 数据面 + 自动回退`”方案。

## 6. 总体架构

### 6.1 架构概览

```mermaid
flowchart LR
    UI["Web UI"] -->|POST /api/local-files/pick| LA["本机 Agent"]
    UI -->|POST /api/transfers/accelerated| LA
    LA -->|HTTPS/TLS 控制面| RA["远端 Agent"]
    LA -->|TCP 数据面 x N| RA
    LA -->|直接 ReadAt 本地文件| LF["本地磁盘文件"]
    RA -->|WriteAt + 预分配| RF["接收端临时文件"]
    RA -->|完成后原子提交| OF["接收目录"]
    LA -.失败回退.->|现有 HTTP/v2-lan-fast| RA
```

### 6.2 分层职责

- `Web UI`：只负责交互、状态展示和发起本地命令。
- `本机 agent 控制面`：负责会话准备、对端协商、回退决策、事件推送。
- `本机 agent 数据面`：负责直接从本地文件读取字节并通过多 TCP 连接发送。
- `远端 agent 控制面`：负责授权接收、资源准备、完成态确认。
- `远端 agent 数据面`：负责会话鉴权、预分配目标文件、按 offset 并发写入。

### 6.3 路由规则

- 文本消息与小文件继续使用现有路径。
- 只有满足“大文件阈值”和“本机支持 native file bridge”两个条件时，UI 才展示或优先走极速模式。
- 高速模式建立失败或中途不可恢复失败时，自动回退到现有文件路径。

## 7. 控制面设计

### 7.1 会话模型

极速大文件传输分为五个阶段：

1. `pick`：发送端本机 agent 通过原生文件对话框获得本地文件句柄来源。
2. `prepare`：发送端通过控制面向接收端申请高速会话。
3. `transfer`：双方建立高速数据面并传输文件。
4. `verify`：接收端校验覆盖完整性与最终摘要。
5. `commit`：成功则提交文件并发出完成事件；失败则回退或终止。

### 7.2 新增本地 API

#### `POST /api/local-files/pick`

用途：

- 由 agent 弹出 Windows 原生文件选择框。
- 只选择单个文件。

返回建议：

```json
{
  "localFileId": "lf_01J...",
  "displayName": "movie.mkv",
  "size": 4294967296,
  "acceleratedEligible": true
}
```

约束：

- UI 只拿到 `localFileId` 和展示信息，不拿到真实路径。
- 真实路径只保存在 agent 内部 lease 表中。

#### `POST /api/transfers/accelerated`

请求建议：

```json
{
  "peerDeviceId": "dev_peer_xxx",
  "localFileId": "lf_01J..."
}
```

返回建议：

```json
{
  "transferId": "tr_01J...",
  "messageId": "msg_01J...",
  "state": "preparing"
}
```

### 7.3 设备间控制面协商

发送端向接收端发起 `accelerated_prepare` 请求，接收端返回：

- `sessionId`
- `transferToken`
- `dataPort`
- `chunkSize`
- `maxStreams`
- `fileSize`
- `fileName`
- `expiresAt`

约束：

- `transferToken` 必须是高熵、一次性、短时效令牌。
- `sessionId` 绑定 `transferId`、发送端设备、接收端设备。
- `dataPort` 来自接收端独立数据监听端口，不复用现有 agent 控制端口。

### 7.4 状态模型

控制面侧建议状态：

- `preparing`
- `ready`
- `transferring`
- `verifying`
- `committing`
- `done`
- `fallback_pending`
- `fallback_transferring`
- `failed`
- `canceled`

要求：

- 同一 `transferId` 在高速路径与回退路径之间保持稳定，避免 UI 出现两条记录。
- 所有进度、速率、错误信息都以 agent 内部事件为单一事实来源。

## 8. 高速数据面设计

### 8.1 基本原则

- 仅承载大文件字节数据，不承载复杂业务语义。
- 使用独立 TCP 数据端口。
- 数据面不走 TLS，但必须依赖控制面下发的短时令牌做会话鉴权。
- 数据面仅接受来自已配对设备、且会话未过期的连接。

### 8.2 连接模型

- 一个高速传输会话对应一个 `sessionId`。
- 一个会话初始先建立 `1` 条数据连接。
- 根据自适应策略逐步扩张到 `2 / 4 / 8` 条连接。
- 每条连接在鉴权成功后持续发送一组分片数据帧。

### 8.3 数据面握手

每条 TCP 连接建立后，发送端先写入固定头 `StreamHello`：

- `magic`
- `version`
- `sessionId`
- `transferToken`
- `streamIndex`

接收端校验成功后返回确认，之后才允许进入数据帧传输阶段。

### 8.4 数据帧协议

V1 使用简单二进制帧，避免 HTTP、multipart 或通用封装带来的额外开销。

建议帧结构：

- `kind`：`DATA / FIN / ABORT`
- `offset`：文件内偏移
- `length`：当前负载长度
- `crc32c`：当前负载校验
- `payload`

设计意图：

- `offset + length` 允许接收端直接 `WriteAt`。
- `crc32c` 用于热路径快速校验，避免 silent corruption。
- 帧头固定、小、易解析，且在大分片条件下头部开销可忽略。

### 8.5 接收模型

接收端在 `prepare` 阶段完成如下动作：

- 为目标文件创建接收任务记录。
- 在临时目录预创建目标文件。
- 按文件总大小执行预分配。
- 初始化分片覆盖位图。

数据面写入规则：

- 每个数据帧校验 `crc32c`。
- 校验成功后按 `offset` 写入预分配文件。
- 写入成功后标记覆盖位图。
- 全部分片覆盖完成后进入 `verifying`。

### 8.6 完整性保证

V1 采用“两层校验”：

1. 热路径：每帧 `crc32c` 校验，快速发现传输或内存数据损坏。
2. 收尾阶段：接收端对已组装完成的文件做一次顺序摘要校验，默认建议使用 `BLAKE3`。

说明：

- 最终摘要复核只发生在接收端，不会让发送端再次把大文件上传到本机 agent。
- 接收端多一次顺序读属于 V1 为了稳定性接受的成本，后续若做断点续传或树哈希，再进一步优化。

## 9. Striping 与自适应策略

### 9.1 为什么不用固定并发度

固定 4 路或固定 8 路不是最优解，因为不同环境的瓶颈不同：

- 千兆有线、2.5G 有线、Wi-Fi 6、机械盘、SSD 的最优并发度不同。
- 发送端磁盘读吞吐、接收端写吞吐、CPU 和网络栈行为都会影响结果。

因此，V1 使用“分档位自适应”而不是固定并发度。

### 9.2 启动条件

- 仅当文件大小达到 `acceleratedThreshold` 时启用高速模式。
- `acceleratedThreshold` 与当前应用内“大文件分流阈值”保持一致来源，V1 默认可与现有 `multipartThreshold` 对齐。

### 9.3 档位集合

并发档位固定集合为：

- `1`
- `2`
- `4`
- `8`

说明：

- 这不是“固定开 8 路”，而是“只在这些稳定档位之间切换”。
- 采用离散档位，便于实现、监控和回退。

### 9.4 自适应算法

初始策略：

- 先使用 `1` 条连接探测。
- 经过预热窗口后，若吞吐增长空间明显，则提升到下一个档位。
- 每次提升后进入新的观察窗口，若收益不足则回退到上一个档位。

建议指标：

- 1 秒滑动窗口吞吐
- 3 秒指数平滑吞吐
- socket 写阻塞时间
- 待发送分片队列深度
- 接收端落盘确认速度

建议判断：

- 升档：最近两个观察窗口吞吐增益持续大于阈值，且发送端读盘与接收端落盘都未明显饱和。
- 降档：升档后吞吐未提升或波动显著恶化，或出现持续写阻塞、接收侧积压。

### 9.5 分片大小

分片大小不固定写死，采用自适应区间，建议范围：

- 最小 `1 MiB`
- 默认 `4 MiB`
- 最大 `8 MiB`

规则：

- 低并发时分片可以更大，减少协议头和调度成本。
- 高并发时可适当减小分片，以改善负载均衡和尾部收敛。
- 分片大小与并发度共同决定单次在途数据量。

## 10. 本地文件桥接与用户流程

### 10.1 设计结论

普通发送与极速发送并存，但入口不同：

- 普通发送：继续浏览器 `input[type=file]` 路径。
- 极速大文件发送：必须通过 native file bridge，让 agent 直接拿到本地文件路径或句柄来源。

### 10.2 LocalFileLease

agent 内部维护：

```text
localFileId -> LocalFileLease
```

建议字段：

- `localFileId`
- `path`
- `displayName`
- `size`
- `modifiedAt`
- `leaseExpiresAt`

可选增强字段：

- Windows 文件唯一标识
- 创建时的基础摘要
- 最近访问时间

规则：

- UI 永远不直接接触真实路径。
- 发送前 agent 必须再次校验文件仍存在且大小未变化。
- lease 过期或文件变化时，要求用户重新选择。

### 10.3 用户流程

极速发送推荐交互：

1. 用户在会话页点击“极速发送大文件”。
2. Web UI 调用 `POST /api/local-files/pick`。
3. agent 弹出 Windows 原生文件选择框。
4. 用户选中文件后，UI 只收到 `localFileId + 文件展示信息`。
5. 用户确认发送，UI 调用 `POST /api/transfers/accelerated`。
6. agent 直接从本地磁盘 `ReadAt` 读取并启动高速会话。
7. UI 通过事件流展示准备中、传输中、速率、进度、回退或完成。

### 10.4 安全边界

- Web UI 不能拿到本机文件真实路径。
- 只有本机 loopback 页面才能调用本地 API。
- native file bridge 不对局域网其他设备开放。

## 11. 鉴权、回退与错误处理

### 11.1 数据面鉴权

每个高速会话的 `transferToken` 必须满足：

- 随机会话令牌，不可预测。
- 与 `sessionId`、发送端设备 ID、接收端设备 ID 绑定。
- 仅在短时间窗口内有效。
- 一旦会话结束立即失效。

### 11.2 自动回退

以下情况触发自动回退到现有 `v2-lan-fast / HTTP` 路径：

- 接收端不支持高速能力。
- native file bridge 不可用。
- 数据端口无法建立连接。
- 高速会话在准备阶段失败。
- 高速数据面在传输早期进入不可恢复错误。

回退要求：

- 保持同一 `transferId` 和 `messageId`。
- UI 不新增重复记录，只更新状态说明。
- 回退过程对用户尽量透明，但必须有系统提示说明已切换到兼容模式。

### 11.3 失败判定

以下情况直接失败，不尝试高速回退重试：

- 文件 lease 失效或源文件已变化。
- 接收端磁盘空间不足。
- 接收端临时目录不可写。
- 对端明确拒收。
- 最终摘要校验失败。

## 12. 模块边界与代码落点

### 12.1 `backend/internal/localfile`

新增职责：

- Windows 原生文件选择桥接
- `LocalFileLease` 生命周期管理
- 文件存在性与元数据复核
- 为应用层暴露可 `ReadAt` 的本地文件访问对象

### 12.2 `backend/internal/api`

新增职责：

- `POST /api/local-files/pick`
- `POST /api/transfers/accelerated`
- 向前端返回极速能力标记
- 将高速传输事件统一推送到现有事件总线

### 12.3 `backend/internal/app`

新增职责：

- `PickLocalFile`
- `SendAcceleratedFile`
- 高速会话与普通会话的分流决策
- 统一 transfer 生命周期、回退策略和 UI 事件语义

### 12.4 `backend/internal/protocol`

新增职责：

- 定义控制面 `accelerated_prepare / accelerated_ready / accelerated_complete / accelerated_abort`
- 定义能力协商字段
- 定义高速会话 metadata 的序列化结构

### 12.5 `backend/internal/transfer`

新增职责：

- 高速数据面监听器
- session sender / receiver 的高速模式实现
- 分片调度
- striping 自适应控制
- 进度、速度、峰值速率统计

说明：

- 现有 `session_sender.go`、`session_receiver.go`、`session_types.go` 可以继续承接公共会话语义。
- 高速数据面细节应尽量拆到独立文件，避免把普通路径和极速路径揉成一个巨型状态机。

### 12.6 `frontend`

新增职责：

- 在支持条件下展示“极速发送大文件”入口
- 展示本地文件已选择状态
- 展示传输速率、进度、回退原因与完成结果
- 保持普通文件发送入口不受影响

## 13. V1 最小可交付范围

V1 范围严格限定为：

- 仅 Windows
- 仅已配对设备
- 仅单个大文件
- 仅发送端本机选文件
- 仅点对点
- 同一时刻优先只保证一个极速任务闭环
- 失败可回退到现有普通传输路径

V1 明确不做：

- 多文件极速队列调度
- 多任务带宽公平分配
- 目录选择
- 断点续传
- 后台长期缓存 lease
- 跨平台原生文件桥接

## 14. 风险与缓解

### 14.1 Windows 原生文件桥接风险

风险：

- 原生文件对话框集成细节多，容易引入线程模型或权限问题。

缓解：

- V1 只做单文件选择。
- 桥接能力单独封装在 `localfile` 模块，不污染传输层。

### 14.2 独立数据端口风险

风险：

- 防火墙、杀毒软件或端口策略可能阻断数据面连接。

缓解：

- 使用固定配置化数据端口。
- 准备阶段显式返回 `dataPort`。
- 连接失败时快速回退到兼容路径。

### 14.3 多连接落盘竞争风险

风险：

- 接收端并发 `WriteAt` 若调度不当，可能造成磁盘抖动。

缓解：

- 通过分片大小和并发度联动控制在途写入量。
- 自适应算法不盲目拉高连接数。

### 14.4 实现复杂度失控风险

风险：

- 若同时引入断点续传、多任务调度、目录发送，会迅速失控。

缓解：

- V1 严格聚焦“单个大文件极速发送”。
- 所有未来变化轴通过清晰模块边界留口，不在本轮一起实现。

## 15. 实现顺序与风险优先级

建议实现顺序如下：

1. 打通控制面会话协商与能力协商，先把高速路径和回退路径的状态机挂到统一 `transferId` 上。
2. 实现接收端独立数据监听、会话鉴权、预分配文件与单连接 `WriteAt` 闭环。
3. 实现发送端 native file bridge 和 `LocalFileLease`，让 agent 直接读取本地文件。
4. 实现单连接高速数据面发送，先验证“非浏览器数据源 + 专用 TCP 端口”带来的吞吐收益。
5. 在单连接稳定后，再引入 `1 -> 2 -> 4 -> 8` 的 striping 自适应。
6. 补齐 UI 入口、进度、速率、回退提示和错误提示。
7. 最后补齐完整性校验、异常测试和回退验收。

排序原则：

- 先验证闭环，再追求极限速度。
- 先保证退路，再激进优化。
- 先拆出边界，再上并发和自适应。

## 16. 验收标准

以下条件同时满足时，认为本设计定义的 V1 达成目标：

- 已配对 Windows 设备之间可从 Web UI 发起极速大文件发送。
- 发送端无需再把本地大文件先上传并落本机临时文件后再发送。
- 高速会话能够显示准备中、传输中、速率、进度、完成或失败。
- 高速数据面失败时能自动回退到现有传输路径，且 UI 不出现重复传输记录。
- 接收端能够正确预分配、写入、校验并提交目标文件。
- 在典型局域网测试环境下，极速模式吞吐显著高于现有普通文件路径。

## 17. 结论

本次设计的核心不是“继续优化浏览器上传”，而是把大文件从浏览器数据源中解耦出来，转为：

- 用现有 HTTPS/TLS 控制面做会话和信任边界
- 用 Windows 原生文件桥接让 agent 直接读取本机磁盘文件
- 用独立明文 TCP 数据面和自适应 striping 压榨局域网带宽
- 用自动回退维持产品闭环和稳定性

这条路线在复杂度、性能、稳定性和交付确定性之间取得了当前版本最合理的平衡。
