# Windows 局域网冒烟矩阵

## 目标

用于验证 V1 在 Windows 局域网中的最小可用链路，包括发现、配对、健康检查、文本消息与文件传输。

## 测试前准备

1. 两台或以上 Windows 设备安装并启动桌面应用。
2. 每台设备都能打开桌面主界面，并看到顶部健康信息层。
3. 每台设备先执行一次 `scripts/smoke-desktop.ps1 -SkipBuild`，确认本机运行目录初始化完成，且主界面已实际加载。
4. 记录设备名称、IP、网络类型、Windows 版本和是否开启 VPN。

## 环境矩阵

| 编号 | 场景 | 环境组合 | 关注点 |
| --- | --- | --- | --- |
| 1 | 基础互通 | Windows 10 + Windows 11，同一路由器 | 自动发现、配对、文本消息 |
| 2 | 多网卡 | WiFi + 有线双网卡 | 设备发现稳定性、连接方向 |
| 3 | VPN 干扰 | 一台开启 VPN | 发现失败提示、手动连接兜底 |
| 4 | 睡眠恢复 | 一台设备睡眠后唤醒 | 在线状态恢复、重连提示 |
| 5 | 防火墙差异 | 一台默认防火墙开启 | 健康检查页诊断、连接失败原因 |
| 6 | 设备重名 | 两台设备配置相同名称 | 列表展示与区分能力 |
| 7 | 大文件边界 | 1GB 文件发送 | 进度、取消、失败重试 |

## 通用检查项

- 设备列表显示远端设备配置的名称。
- 首次配对时，双方都能看到相同的 6 位短码。
- 顶部健康信息层能看到监听端口、自动发现状态和当前问题提示。
- 对方离线时，发送入口明确提示不可发送。
- 文件失败、取消、拒收后，状态与提示一致。

## 场景执行要求

### 1. 基础互通

1. 观察双方是否自动发现彼此。
2. 发起首次配对并核对短码。
3. 发送一条文本消息和一个小文件。

预期结果：

- 双方均能完成配对。
- 文本消息实时到达。
- 文件状态按发送中、校验中、完成推进。

### 2. 多网卡

1. 让一台设备同时启用 WiFi 和有线网络。
2. 观察另一台设备的发现结果与连接结果。

预期结果：

- 设备列表不会重复刷出同一设备。
- 若发现成功但连接失败，顶部健康信息层可定位最近错误。

### 3. VPN 干扰

1. 在一台设备开启 VPN。
2. 观察自动发现是否受影响。
3. 改用手动 `IP:port` 连接。

预期结果：

- 自动发现失败时有明确提示。
- 手动连接路径仍可用于配对与发送。

### 4. 睡眠恢复

1. 配对成功后让接收端进入睡眠。
2. 唤醒设备并等待网络恢复。

预期结果：

- 会话历史仍可查看。
- 在线状态能恢复，失败中的发送不会误报成功。

### 5. 防火墙差异

1. 保持一台设备的默认防火墙规则不变。
2. 发起发现、配对和消息发送。

预期结果：

- 顶部健康信息层能提示端口或连接问题。
- 连接失败原因可导出到诊断结果。

### 6. 设备重名

1. 在两台设备上设置相同设备名称。
2. 观察发现页与设备列表。

预期结果：

- 设备列表仍显示配置名称。
- 用户仍能基于其他信息完成正确配对。

### 7. 大文件边界

1. 发送一个约 1GB 的测试文件。
2. 过程中执行取消、重新发送和网络抖动模拟。

预期结果：

- 进度持续更新。
- 取消后状态收敛到已取消。
- 失败后允许重试，且不会产生误报成功。
