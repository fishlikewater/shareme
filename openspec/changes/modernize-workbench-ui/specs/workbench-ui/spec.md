## ADDED Requirements

### Requirement: 系统必须提供现代清爽的工作台式主布局

系统 MUST 将前端主界面组织为紧凑应用栏、设备 Dock、会话主工作区和状态区域，而不是以大 hero 展示区作为首屏主体。该布局 MUST 在 Wails 桌面入口和 localhost Web UI 入口中保持一致的核心操作路径。

#### Scenario: 用户进入界面后直接看到可操作工作区
- **WHEN** 前端完成启动快照加载
- **THEN** 系统必须优先展示设备切换入口、当前设备上下文、消息工作区和发送入口，而不是以大标题或产品展示内容占据首屏主体

#### Scenario: 两种宿主入口共享同一工作台语义
- **WHEN** 用户分别通过 Wails 桌面入口和 localhost Web UI 入口打开界面
- **THEN** 系统必须呈现一致的设备切换、会话、配对提示和发送入口语义

### Requirement: 系统必须提供可折叠设备 Dock

系统 MUST 使用可折叠设备 Dock 展示已发现设备。设备 Dock 在展开态 MUST 显示设备名称、配对状态、可达状态和最近消息摘要；在收起态 MUST 保留足够信息让用户识别设备、判断状态并切换会话。

#### Scenario: 展开态展示完整设备状态
- **WHEN** 设备 Dock 处于展开状态且存在已发现设备
- **THEN** 系统必须为每台设备展示完整设备名称、是否已配对、是否可达以及最近消息或状态提示

#### Scenario: 收起态仍可判断设备状态
- **WHEN** 设备 Dock 处于收起状态且存在已发现设备
- **THEN** 系统必须为每台设备保留可识别短名或标识、选中态、可达/不可达状态提示以及配对/待配对提示

#### Scenario: 用户可以切换折叠状态
- **WHEN** 用户触发设备 Dock 的折叠或展开控制
- **THEN** 系统必须在不重新加载启动快照、不改变已选设备、不改变后端数据的情况下更新本地布局状态

### Requirement: 系统必须强化设备切换与状态判断

系统 MUST 让设备项的选中态、配对状态、可达状态和可发送状态在视觉和文本上同时可辨认。状态表达 MUST NOT 只依赖颜色。

#### Scenario: 当前设备选中态可被视觉和辅助技术识别
- **WHEN** 用户选中一台设备
- **THEN** 该设备项必须呈现明确选中态，并继续通过 `aria-pressed` 或等价语义暴露给辅助技术

#### Scenario: 不同设备状态可被文本识别
- **WHEN** 设备处于已配对可达、待配对、已配对但不可达任一状态
- **THEN** 系统必须展示对应文本提示或等价可访问名称，不能只用颜色或图形区分状态

### Requirement: 系统必须将会话操作集中在主工作区

系统 MUST 在主工作区内展示当前设备上下文、消息流、发送器、配对引导和不可达提示。未配对或不可达状态 MUST 阻止不可用发送操作，同时保留用户理解下一步所需的信息。

#### Scenario: 未配对设备显示信任建立路径
- **WHEN** 用户选中未配对设备
- **THEN** 系统必须在主工作区显示配对引导或短码确认入口，并禁用或隐藏不可用发送操作

#### Scenario: 不可达设备保留历史查看并阻止发送
- **WHEN** 用户选中已配对但当前不可达的设备
- **THEN** 系统必须允许用户查看既有消息上下文，并显示不可达提示且阻止当前发送操作

#### Scenario: 可发送设备显示完整发送入口
- **WHEN** 用户选中已配对且可达设备
- **THEN** 系统必须在主工作区显示文字发送、普通文件发送和大文件发送相关入口

### Requirement: 系统必须以克制状态区域呈现健康与传输信息

系统 MUST 将健康状态和传输状态呈现为不干扰主工作区的状态区域。正常状态下该区域 SHOULD 保持摘要化；当存在错误、警告或活跃传输时，系统 MUST 提升对应信息可见性。

#### Scenario: 正常状态不挤压主会话空间
- **WHEN** 系统健康状态正常且没有活跃传输
- **THEN** 状态区域必须以摘要方式呈现，不得显著挤压消息流和发送器空间

#### Scenario: 活跃传输显示可见摘要
- **WHEN** 存在活跃文件传输
- **THEN** 系统必须展示传输摘要和进度入口，使用户无需离开当前会话即可确认传输状态

#### Scenario: 异常状态提升可见性
- **WHEN** 健康状态为警告或错误
- **THEN** 系统必须提升异常提示的视觉优先级，并展示可理解的状态说明

### Requirement: 系统必须满足响应式与可访问性约束

系统 MUST 在窄屏、平板和桌面宽度下保持可用，不产生非预期横向滚动。交互控件 MUST 具备稳定尺寸、可见焦点态和清晰可访问名称。

#### Scenario: 关键宽度下无横向滚动
- **WHEN** 用户在 375px、768px、1024px 或 1440px 宽度查看界面
- **THEN** 系统必须保持主操作路径可见且不得出现由布局溢出导致的页面横向滚动

#### Scenario: 键盘用户可以识别焦点
- **WHEN** 用户通过键盘 Tab 导航设备项、折叠按钮、发送按钮和输入框
- **THEN** 系统必须展示可见焦点态，并保持焦点顺序与视觉阅读顺序一致

#### Scenario: 降低动效设置被尊重
- **WHEN** 用户系统启用 `prefers-reduced-motion`
- **THEN** 系统必须禁用或显著减少侧栏折叠、状态更新和 hover 的非必要动画
