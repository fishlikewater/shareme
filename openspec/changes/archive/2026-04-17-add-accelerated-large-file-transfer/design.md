## Context

当前 Message Share 已具备点对点文本和文件发送能力，Web UI 通过本机 HTTP/WebSocket 调用本机 agent。现有文件发送链路以浏览器文件输入和通用 HTTP 传输为主，适合普通文件发送，但会在“局域网内单个大文件吃满带宽”这个目标上引入额外用户态搬运、通用协议开销和浏览器侧限制。

本次变化面向局域网、已配对设备、点对点单大文件场景。约束包括：

- 不破坏现有文本消息和普通文件发送路径
- 不引入账号、离线消息、跨网段穿透和断点续传
- V1 只保证 Windows 原生文件桥接
- 失败时必须回到已有传输链路，保证产品闭环

## Goals / Non-Goals

**Goals:**

- 为单个大文件提供接近局域网链路上限的峰值吞吐
- 避免发送端把本地大文件再次经过浏览器上传到本机 agent
- 将高速路径纳入统一 `transferId` 状态机和 Web UI 事件流
- 为后续群发、公告板广播、断点续传保留清晰模块边界

**Non-Goals:**

- 不在本次设计中实现目录发送、多文件极速调度和断点续传
- 不以 UDP/QUIC 自研协议为本轮目标
- 不在本轮保证 Linux/macOS 原生文件选择桥接
- 不改变现有普通小文件发送入口

## Decisions

### 决策 1：采用“控制面保留现有安全链路 + 数据面新增专用高速通道”

选择：

- 控制面继续沿用现有 agent 间安全控制链路做配对、鉴权、协商和事件语义
- 大文件数据面新增独立端口，不复用现有控制通道

原因：

- 现有控制面已经承载信任关系、会话语义和错误回传，继续复用可以减少系统重写范围
- 把大字节流从控制面剥离，能避免控制链路被高吞吐流量挤压

备选方案：

- 继续优化现有 HTTP 文件路径：改动小，但无法解决浏览器与通用协议开销
- 全量重做为单通道新协议：理论上更纯粹，但迁移和回退成本过高

### 决策 2：高速数据面使用专用明文 TCP，而不是 TLS、UDP 或 QUIC

选择：

- 数据面采用专用明文 TCP
- 由控制面下发一次性短时效 `transferToken` 做会话鉴权

原因：

- 局域网已配对设备之间，大文件数据面最主要目标是吞吐而非公网暴露下的防窃听
- TCP 能复用成熟的可靠传输、流控和拥塞控制，复杂度远低于 UDP/QUIC 自研
- 去掉 TLS 能减少 CPU 和协议封装开销，为高速模式争取更多吞吐空间

备选方案：

- TLS over TCP：安全边界更统一，但会增加 CPU 成本，不利于 V1 极限吞吐目标
- UDP/QUIC：可调空间更大，但复杂度、调试成本和环境不确定性都更高

### 决策 3：极速发送不再让浏览器成为大文件数据源

选择：

- 普通文件继续用浏览器 `input[type=file]`
- 极速大文件模式改为由 agent 弹出 Windows 原生文件选择框，并维护 `LocalFileLease`

原因：

- 用户选中的文件本来就在本机磁盘，浏览器再把数据上传给本机 agent 属于重复搬运
- agent 直接获得本地文件来源后，可以使用 `ReadAt` 和多连接 striping 做高吞吐发送
- 通过 `localFileId` 间接引用文件，可以避免向 Web UI 暴露真实本地路径

备选方案：

- 继续使用浏览器文件对象：实现最简单，但无法消除发送端多一跳的数据搬运
- 让前端上传到本机临时文件再发送：I/O 最浪费，不符合本次目标

### 决策 4：使用离散档位的 striping 自适应，而不是固定并发度

选择：

- 会话先用 1 条 TCP 连接探测
- 根据实时吞吐和阻塞情况，逐步切换到 `2 / 4 / 8` 条连接

原因：

- 局域网环境差异大，固定 4 路或 8 路无法同时覆盖千兆、2.5G、Wi-Fi 6、SSD、机械盘等差异
- 离散档位实现比连续调参更稳定，也更容易监控和回退

备选方案：

- 固定 4 路：简单，但在不同环境下可能不足或过度
- 连续自适应任意路数：更灵活，但实现和调试复杂度更高

### 决策 5：接收端预分配文件并按 offset 并发 WriteAt

选择：

- 接收端在准备阶段创建临时文件并预分配目标大小
- 数据面按 `offset + length` 接收，直接并发 `WriteAt`

原因：

- 可避免把多路连接重新串行拼接成单流后再写盘
- 有利于发送端将多个连接稳定映射到文件区间，提高总吞吐
- 配合覆盖位图和最终摘要校验，能明确判断完整性

备选方案：

- 每路先写独立分片文件再合并：磁盘 I/O 更高，尾部合并成本大
- 单写线程串行落盘：实现简单，但会更早碰到单线程写入瓶颈

### 决策 6：高速失败时自动回退到现有传输路径，且保留同一 transfer 语义

选择：

- 高速路径失败时自动切换到现有普通文件传输路径
- 维持同一 `transferId` 和 `messageId`

原因：

- V1 的关键是可交付，而不是只在理想环境下快
- 统一 transfer 语义可以让 UI、存储和日志保持单记录闭环

备选方案：

- 失败后直接报错：实现简单，但用户体验差，且不利于产品落地
- 重新创建一条新传输记录：会让 UI 和状态管理更混乱

## Risks / Trade-offs

- [Windows 原生文件桥接只做单平台] → 通过独立 `localfile` 模块隔离平台依赖，避免污染传输层
- [明文 TCP 数据面降低了链路加密强度] → 仅在已配对局域网设备之间启用，并要求短时效会话令牌和独立数据端口
- [多连接并发写盘可能放大磁盘抖动] → 通过离散档位并发和分片大小联动，优先保证稳定增益
- [高速模式新增了更多状态和失败路径] → 所有状态统一回流到现有 transfer 事件模型，不额外引入第二套 UI 语义
- [本轮范围若继续扩张到目录、多任务、断点续传，会迅速失控] → V1 严格限定为已配对设备之间的单个大文件极速发送

## Migration Plan

1. 先保留现有普通文件发送链路不变，新增高速能力协商与后端入口
2. 仅在满足“已配对 + 文件达到阈值 + 本机支持原生文件桥接 + 对端支持高速能力”时启用高速模式
3. Web UI 在高速模式不可用时仍保留普通文件发送能力
4. 若上线后发现数据面兼容性或稳定性问题，可通过禁用高速能力协商和入口展示回退到现有普通路径
5. 回滚不涉及历史数据迁移，旧版本仍可继续处理普通文件传输

## Open Questions
